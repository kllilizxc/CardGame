# 调整为成熟的游戏架构

## Objective

当前我们已经添加了主要的功能部分，但是不确定现在的代码架构是否成熟，是否能够承担后续大量剧本，卡片资源，地图资源等的开发，所以要在这个时候先调整好。比如是否能方便的用配置文件来描述地图/关卡/卡牌资源等，角色的属性是否有统一的配置地方，属性的效果是否可灵活配置修改，是否便于保存/加载功能，等等。

Clarified planner intent on 2026-05-09: this Goal is an architecture-spine goal, not a request for a broad content push or a cosmetic cleanup. The project should be able to grow toward a medium-to-large 2D game where ordinary content expansion is data/config driven, runtime ownership is explicit, and save/load remains durable as more story, card, map, character, and world-state content is added.

## Success Criteria

讲代码架构调整为可以简单 scale 的模式，以成熟的中大型 2d游戏为目标

Refined success criteria:

- Content authors can add common stories, Hub locations, world-map destinations, Expedition maps/levels, encounters, cards, statuses, and character/world data through versioned data contracts instead of editing scene code for each new entry.
- Runtime domains have clear owners and interfaces: content loading/validation, scene routing, battle/effect resolution, world/progression state, persistence/save-load, and presentation/UI.
- Game data uses stable IDs and cross-resource references that can be validated by focused tests or tools before runtime.
- Player/character attributes, status/effect vocabulary, rewards, and world flags have a unified model or an explicitly documented migration path toward one.
- Save/load can eventually restore world-map, Hub, Story, Expedition, stash/reward, and player progression state through versioned boundaries rather than scattered ad hoc local-storage formats.
- Refactors stay incremental: every slice keeps the checked-in game playable and tests/build green; no big-bang rewrite unless a later audit proves it unavoidable.

## Current Strategy

- Continue the incremental architecture-spine migration rather than a big-bang rewrite: harden one durable content/runtime/save boundary at a time, keep the game playable, and only promote small independently reviewable implementation slices.
- Current approach remains the incremental contract spine: versioned content/catalog IDs, cross-resource validation, source-aware launch/data-loading ownership, read/write-compatible save/world-state boundaries, and shared stat/effect vocabulary alignment through adapters rather than a replacement combat engine.
- 2026-05-10 planner refresh: injected persistent-stash writes (`65098e8`), the gongfa action dispatch registry (`e56db05`), and manual JSON save-document text transfer (`9617669`) have landed. The current generator batch keeps three disjoint, behavior-preserving tracks: a narrow `GameWorldState` persistent-stash write boundary, pure gongfa condition evaluation extraction, and a content-catalog validation-index split.

## Current Focus

- Work on a narrow `GameWorldState` persistent-stash write boundary that builds on the read-only projection, pure stash operations, and injected `savePersistentStash` without replacing current Expedition/RunResolution runtime writers yet.
- Work on a pure gongfa condition-evaluation helper so `UnitEffectManager` delegates both action dispatch and condition checks through focused typed seams while preserving existing warnings and combat behavior.
- Work on a content catalog validation-index split to reduce the monolithic catalog module without changing catalog rules, runtime resolver behavior, checked-in content, scenes, save behavior, or UI.
- Keep unsupported gongfa actions, universal effect interpretation, combat balance changes, new save keys, real save migrations, browser restore/import-export UI, scene-triggered save/restore flows, broad world/player-state write ownership, economy redesign, broad scene rewrites, product content/UI, authoring tooling, and release/deploy work deferred until their prerequisite reviews or seams are accepted.

### Highest-risk scaling bottlenecks

1. **Scene-owned content loading without a manifest.** Resource paths and cache keys are spread across WorldMap, Hub, Story, Expedition, Battle, and `StatusHelper`; adding many files will make resource ownership and schema-version drift hard to see.
2. **Missing global cross-resource validation.** Existing validators catch local shape and some intra-file refs, but there is no single checked command that validates WorldMap -> Hub/Expedition files and IDs, Hub -> Story graphs, Story/Expedition -> encounters/decks, deck/reward/encounter -> card IDs, card -> gongfa IDs, or status IDs.
3. **Large runtime classes are accumulating multiple responsibilities.** A 2026-05-09 line-count check shows `BattleAnimationManager.ts` at 1203 LOC, `BattleScene.ts` at 991 LOC, `ExpeditionScene.ts` at 883 LOC, `storyFlowViewModel.ts` at 673 LOC, and `UnitEffectManager.ts` at 655 LOC. Size alone is not a bug, but these files combine data loading, orchestration, UI, and semantic decisions in ways that will slow safe migration.
4. **Effect/stat vocabulary is fragmented.** Story uses `StoryCondition`/`StoryEffect`; card data has legacy `effects`; gongfa has its own schema/action enum; statuses have config definitions and runtime instances; Expedition rewards use card/item/spirit-stone bundles. Without shared IDs/operations, new content can express similar mechanics in incompatible ways.
5. **Battle semantic dispatch has inconsistent seams.** `UnitEffectManager` implements only a subset of configured action types, warns for several others, evaluates expressions with `new Function`, and coexists with `SkillEffectHandler`, `GameActionHandler`, card-type managers, `StatusManager`, and animation-only `EffectManager`. This raises regression risk for richer card/gongfa/status content.
6. **Persistence is split before world-state ownership exists.** Story/Hub persistence is versioned, and Expedition active runs are route-keyed, but the persistent stash remains a separate global key and there is no save document that owns world-map resume state, player progression, story snapshots, active expeditions, stash, unlocks, and content-version metadata together.
7. **Authoring scale depends on string paths and IDs.** The checked-in data is already data-driven, but many route-critical refs are raw strings in JSON and tests. As content volume grows, missing files, ID mismatches, and stale test fixtures become likely without a catalog and validation harness.

### Existing patterns worth preserving

- **Data-driven route ownership:** `public/data/world/world-map.json` already owns Hub/Expedition target identity, target files, optional Hub location, status copy, and map-marker presentation.
- **Pure contract modules with focused tests:** `worldMap.ts`, `hubTown.ts`, `storyFlow.ts`, `storyFlowViewModel.ts`, `storyBattleRoundTrip.ts`, `expeditionSceneLaunch.ts`, `mapTraversal.ts`, `nonCombatNodeFlow.ts`, `runResultFlow.ts`, and `RunResolution.ts` isolate many rules outside Phaser rendering.
- **Source-aware launch contracts:** Story battle payloads and Expedition battle payloads stay separate, and `BattleScene.handleBattleEnd()` routes completion back to `StoryScene` or `ExpeditionScene` based on the launch source instead of spoofing one runtime as another.
- **Versioned and keyed persistence slices:** `StoryHubSessionPersistence` uses a schema-versioned Story/Hub session document, and Expedition active runs are partitioned by normalized `expeditionId + mapId` route identity while preserving direct/default launch fallbacks.
- **Incremental playable-loop discipline:** README and tests preserve `Boot -> Preloader -> MainMenu -> WorldMapScene -> HubScene | ExpeditionScene`, with `BattleScene` reused by Story and Expedition. This should remain the migration constraint.
- **Checked-in content tests:** The current focused tests already exercise route normalization, Hub navigation/session identity, Story battle round-trips, Expedition route keys, run resolution, and prototype content validation. New architecture seams should extend these rather than bypass them.

### Candidate first migration seams from this audit

1. **Read-only content catalog plus cross-resource validation — recommended first.** Add a versioned manifest for checked-in resources and register existing validators behind it. Validate file existence, declared IDs, WorldMap route targets, Hub story targets, Story battle encounter/deck refs, Expedition map/event/shop/encounter refs, deck/card refs, card `gongfaIds`, status IDs, and reward item/card refs. This is low behavior risk because scenes can keep their current `this.load.json` paths while tests/tooling harden authoring.
2. **Versioned save/world-state compatibility boundary.** Wrap `StoryHubSessionPersistence`, route-keyed active runs, and global stash behind a future save service with schema/migration hooks. This should follow catalog identity work so save metadata can reference content versions/resources.
3. **Shared stat/effect vocabulary and adapters.** Define stable stat/resource/status/effect operation IDs across Story, card/gongfa, status, reward, and world-state data. Start with registries and validators, not a replacement combat interpreter.
4. **Launch/data-loading adapter seam.** After the catalog is green, let route payloads carry catalog resource IDs while preserving current path fields as compatibility aliases. WorldMap-to-Hub/Expedition is the safest first route family because route payload ownership is already centralized.
5. **Scene/controller decomposition.** Extract data-loading and orchestration controllers from `BattleScene` and `ExpeditionScene` only after contracts are stable enough to prevent speculative rewrites.
6. **Battle effect resolver normalization.** Consolidate card/gongfa/skill/status semantic dispatch after vocabulary validation exists. This has high payoff but should not be first because it touches combat behavior and multiple partial implementations.


## Selected Target Architecture Direction (2026-05-09 Generator)

The selected direction is an **incremental contract spine**, not a replacement rewrite. The game should grow around a small set of versioned contracts that are validated outside Phaser scenes first, then adopted by runtime systems one seam at a time. Existing playable routes (`MainMenu -> WorldMapScene -> HubScene | ExpeditionScene -> StoryScene | BattleScene`) remain valid during every migration stage.

### Ownership boundaries

1. **Content catalog and schema ownership**
   - Introduce a versioned content catalog/manifest as the owner of resource identity, resource kind, schema version, file path, and parser/validator ownership.
   - Content JSON files continue to own their domain IDs (`storyId`, `hubId`, `mapId`, `cardId`, `statusId`, etc.); the catalog owns where those resources live and which contract validates them.
   - Existing domain validators such as `worldMap.ts`, `hubTown.ts`, `storyFlow.ts`, `storyContent.ts`, and `prototypeExpeditionContent.ts` should be preserved and registered behind the catalog instead of being replaced wholesale.
   - Scenes should eventually request content by stable resource identity or launch target identity, not by hardcoded file path. File-path fields can remain as compatibility aliases until the catalog owns all checked-in content.

2. **Cross-resource reference validation**
   - Add a focused validation layer that loads catalog resources and checks references across resource boundaries before runtime: world-map destination files, Hub `targetLocationId` and `startStory.storyGraphFile`, Story graph node/battle continuation IDs, Story battle encounter/deck files, Expedition `mapId` and node payload refs, event/shop node refs, deck/card refs, card `gongfaIds`, status IDs, reward card/item IDs, and encounter unit card IDs.
   - Validation should fail in tests/tooling with actionable messages before a scene is opened. Runtime scenes may still show safe errors, but they should not be the first line of defense for authoring mistakes.

3. **Scene launch and data-loading contracts**
   - Keep source-aware launch contracts as the durable routing model: WorldMap owns route targets; Hub owns Hub action/story launch; Story battle and Expedition battle payloads remain separate and must not spoof each other.
   - Normalize launches in pure modules before scenes render. Phaser scenes should orchestrate cache loading, scene transitions, and presentation; pure contract modules own launch defaults, route keys, and data validation.
   - Move toward a shared `ContentLoader`/`ContentCatalog` seam that returns parsed domain resources, while preserving current Phaser `this.load.json` usage through adapters during migration.

4. **Unified player/world-state ownership**
   - Introduce a single `GameWorldState` / `PlayerState` ownership boundary for durable player progression, world flags, location/session state, stash/inventory, active expeditions, story snapshots, unlocks, and run summaries.
   - Existing `StoryHubSessionPersistence` and `RunPersistence` are useful vertical-slice boundaries and should be wrapped/migrated, not deleted first. Until the unified boundary is ready, they remain the only owners of their current localStorage keys.
   - No scene should gain new direct localStorage access. New persistence writes go through the state/save service boundary.

5. **Save/load versioning**
   - The target save format is a versioned local save document with explicit `schemaVersion`, content/catalog version metadata, domain slices, and ordered migration functions.
   - Loading should parse, migrate, clone, and validate the save document before exposing domain state. Corrupt or unsupported slices should fail safely or reset only the affected slice when possible.
   - Existing keys (`cardgame.story-hub-session.v1`, `cardgame.active-run.v1:*`, and `cardgame.persistent-stash.v1`) should be read through compatibility adapters until a migration has proven it can preserve current Hub, Story, Expedition, and stash behavior.

6. **Shared stat/effect vocabulary**
   - Define a shared vocabulary for stat IDs, resources, flags, relations, statuses, effect operations, conditions, targets, and numeric modifiers that can be used by Story, cards/gongfa, statuses, encounters, rewards, and player/world state.
   - Preserve existing Story effects and battle/gongfa/status systems behind adapters first. The first goal is vocabulary alignment and validation, not a new universal effect interpreter that rewrites combat.
   - `Custom`/script-like effect escape hatches should remain quarantined and should not become the default content authoring path.

### Staged migration order that keeps the game playable

1. **Baseline and contract inventory** — Record the current content files, IDs, launch payloads, storage keys, and green verification commands. Do not change behavior.
2. **Read-only content catalog** — Add a catalog/manifest for checked-in WorldMap, Hub, Story, Expedition, card, deck, encounter, status, gongfa, and config resources. Register existing validators and assert that all current content still loads. Scenes keep using their current paths.
3. **Cross-resource validation tests** — Add validation coverage for the highest-risk references first: world-map route targets, Hub actions/story files, Story graph battle refs, Expedition map/event/shop refs, decks/cards/encounters/status/gongfa IDs. Fix data inconsistencies without adding product scope.
4. **Launch/data-loading adapter seam** — Let route payloads carry catalog resource IDs while still accepting current file fields. Add a loader adapter that resolves IDs to existing files and parsed resources. Update one route family at a time, starting with WorldMap-to-Hub/Expedition because those launch contracts already centralize target ownership.
5. **Versioned save service compatibility layer** — Introduce the versioned save document and migration hooks while reading current localStorage keys through adapters. Initially mirror or wrap existing Story/Hub and Expedition persistence so save/load behavior remains unchanged.
6. **Unified world/player-state migration** — Move Hub session, Story runtime snapshots, Expedition active runs, persistent stash, world-map resume/position, unlock flags, and player progression into the unified state boundary one slice at a time. Remove old direct storage only after migration tests prove compatibility.
7. **Stat/effect vocabulary alignment** — Add shared registries and adapters for Story attributes/flags, card stats, statuses, gongfa actions, rewards, and resources. Convert one low-risk operation family at a time while keeping BattleScene's current resolver behavior intact.
8. **Authoring workflow hardening** — Update author docs/templates and require catalog validation before broad content expansion. Only after the contract spine is stable should the project add large story/map/card batches or richer authoring tools.

## Brainstorming Notes

### Approaches considered

1. **Big-bang architecture rewrite** — could impose a clean structure quickly, but it risks breaking the playable prototype, hiding regressions, and forcing speculative abstractions before the project knows which seams are real.
2. **Feature-led continuation only** — keeps shipping content, but it leaves the project exposed to duplicated data contracts, scattered persistence, hard-to-validate references, and scene-level coupling as content volume grows.
3. **Incremental architecture spine** — audit current seams, write target contracts, and migrate one foundation at a time while preserving the playable loop. This is the recommended long-term-safe path.

### First iteration shape

The first iteration should be planning/design heavy and code-light until the target seams are explicit:

1. Map the current architecture and scale risks using evidence from `src/game/**`, `public/data/**`, README, and existing tests.
2. Define the target content/runtime architecture and migration roadmap.
3. Convert only the first selected seam into implementation work after the design is reviewed.
