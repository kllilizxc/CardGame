# 故事系统

## Objective

添加一个故事系统，用于平时非战斗场景，首先有一个地点（比如 xx 城），里面有多个小地点（如铁匠铺等），然后有多个选项（对话，操作等）。需要用多个 json 来描述故事对话支线，支线有触发条件（比如某属性大于 xx，xx对话出发过，或者多种结合）。

### Clarified Intent

故事系统要作为“非战斗 / 平时场景”的通用叙事层，而不是秘境事件节点的专用补丁。第一轮目标是做出一个可验证的垂直切片：

- 用 JSON 描述“地点 → 小地点 → 可选互动 / 对话节点 → 选择 → 后续节点”的故事图。
- 选择项支持声明式触发条件：玩家属性阈值、故事 flag、是否访问过某节点 / 触发过某对话，以及 `all / any / not` 组合。
- 选择项支持声明式效果：设置 flag、记录访问 / 对话、修改基础数值或关系、跳转到下一个故事节点。
- 提供一个小型示例故事（建议复用现有青云宗山门 / 城镇语境），用于证明数据、运行时和 UI 可以连起来。
- 与现有战斗 / 秘境系统共存；第一轮不重写 `BattleScene`，也不把所有秘境非战斗事件迁移到故事系统。

## Success Criteria

所有故事相关的功能。外带一个示例故事。

### Refined Success Criteria

- 有清晰的故事 JSON 内容契约、TypeScript 类型和 dev-time 校验，能拒绝缺失节点、坏跳转、未知条件 / 效果等错误内容。
- 有独立的故事运行态：当前位置 / 小地点、已访问节点、已触发对话、story flags、关系或基础属性快照；第一轮可使用本地持久化或现有 world state seed。
- 有可测试的条件与效果引擎，覆盖属性比较、flag 检查、历史检查，以及组合条件。
- 有最小 Phaser 故事界面 / 场景，能加载示例 JSON，显示地点、小地点、对话 / 操作选项，隐藏或禁用不满足条件的选项，并在点击后应用效果与跳转。
- 示例故事至少包含 1 个大地点、2 个小地点、3 个以上互动选项、1 条带属性条件的分支、1 条依赖前置对话或 flag 的分支。
- 提供轻量测试或构建检查，证明 JSON 校验、条件效果、导航流程不会回归。

- Continue this Goal one narrow product increment at a time. Do not treat the original first-iteration success criteria as a reason to archive the Goal, but also do not promote broad “all story features” work.

The next implementation task must wait for one human product choice instead of guessing between content expansion, unlock/world-state ownership, current-position/map-state behavior, Hub operations, Expedition/story integration, or content-rich secret-realm work.
- Planner recommendation: keep the Goal `blocked` until the human chooses exactly one post-sub-map increment; then promote 1-2 small independent tasks whose files and acceptance checks do not collide.

### Approaches Considered

1. **推荐：通用故事运行时 + JSON 内容契约**
   - 优点：可用于城镇、门派、支线、Hub；可测试；示例故事能逐步扩展为主线。
   - 代价：第一轮需要先做模型 / 校验 / 状态层，UI 需要保持最小。
2. **复用秘境 event / shop 节点格式**
   - 优点：接近已有代码，短期快。
   - 缺点：当前格式偏“随机事件 / 商店奖励”，不适合多地点、多轮对话、前置条件和长期 story flags。
3. **把示例剧情直接写入 Phaser Scene**
   - 优点：最快看到界面。
   - 缺点：无法满足“多个 JSON 描述支线”和条件触发需求，后续维护成本高。
